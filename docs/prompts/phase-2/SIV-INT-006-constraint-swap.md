## Objective

Extend the intervention core with INT-02. Select a task-authored, human-reviewed alternative constraint for the exact target step, replace only the step’s `constraint`, run the existing perturbed execution path, and persist the complete paired baseline/INT-02 evidence. Create the missing isolated SIEVE-T3 refactor-under-constraint fixture required by the Phase 2 DoD, then complete the Phase 2 cross-product: INT-01 and INT-02 each run on SIEVE-T1 and SIEVE-T3. The fixture is acceptance infrastructure for this requirement, not a new requirement ID or a harness change.

## Interface specification

```python
# src/sieve/interventions.py
class ConstraintSwap:
    type: Literal["INT-02"] = "INT-02"
    target_field: Literal["constraint"] = "constraint"

    def __init__(self, alternative_constraints: Mapping[str, str]) -> None:
        """Map target step IDs to non-empty, task-authored alternative constraints."""

    @classmethod
    def from_task_fixture(cls, task_dir: Path) -> "ConstraintSwap":
        """Load task_dir / 'intervention_constraints.json'."""

    def edit(self, baseline_step: StructuredReasoningStep) -> StructuredReasoningStep:
        """Replace only baseline_step.constraint using its exact step ID."""

    def metadata(self, baseline_step: StructuredReasoningStep) -> InterventionMetadata:
        """Return complete INT-02 metadata using the selected alternative."""

# Existing InterventionRunner.run accepts ClaimDeletion | ConstraintSwap.
# src/sieve/cli.py
# `sieve intervene --type` choices are exactly INT-01 and INT-02. For INT-02,
# load ConstraintSwap.from_task_fixture(root / 'tasks' / baseline.task_id).
```

The constraint fixture files must each have this exact shape, with their task-specific step ID, alternative, and review text:

```json
{
  "constraint_swaps": {
    "TSIEVE-T3-S001": "Preserve a leading # tag for channel-style identifiers during the refactor."
  },
  "manual_reviews": {
    "TSIEVE-T3-S001": "A # tag is a concrete, plausible competing identifier-preservation rule; it conflicts with the original @ requirement and is intentionally used as the counterfactual."
  }
}
```

`ConstraintSwap.from_task_fixture` rejects absent files, invalid JSON, a missing `constraint_swaps` or `manual_reviews` object, a non-string alternative or review, an empty or whitespace-only alternative or review, an alternative equal to the baseline constraint, and a target step that has no alternative and review. It must never invent replacement wording. The stored alternative and review are the manually reviewed fixture set.

## Implementation notes

The binding mechanism is: “Replace `constraint` with a plausible but different one; resume.” The only altered target field is `constraint`. Preserve `claim`, `hypothesis`, action fields, target ID, and success criterion exactly; the replacement must be non-empty and differ from the original. Do not blank a constraint, use a generic fallback such as “use another approach,” or ask the live model to author the alternative.

SIEVE-T3 is the §9 “Refactor under constraint” fixture. Its baseline instruction is to extract a private username normalizer while preserving a leading `@` and the public `formatUsername(value: string): string` API. Its recorded INT-02 alternative substitutes only the plausible, conflicting `#`-tag preservation constraint. Keep that alternative and its manual review in `intervention_constraints.json`, not in `task.md`, because the baseline agent receives `task.md` as its task prompt. The SIEVE-T1 mapping similarly substitutes only its `Anonymous` fallback constraint. The baseline tests deliberately remain their original acceptance oracles, so both recorded INT-02 results are allowed and expected to fail. The recorded T3 INT-01 run proves the existing ClaimDeletion path applies to the newly created fixture. “Run successfully” in the Phase 2 DoD means the intervention execution terminates and persists valid evidence; it does not mean every counterfactual task test passes.
